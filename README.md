## Запуск приложения

### Способ 1: IntelliJ IDEA (рекомендуемый)

1. Откройте проект в IntelliJ IDEA
2. В дереве проекта перейдите в `src/main/java/org/example/Main.java`
3. Нажмите правой кнопкой мыши по файлу `Main.java`
4. Выберите **Run 'Main.main()'**

### Способ 2: Командная строка (Windows / Linux / Mac)

```
cd library-manager

javac -d out src/main/java/org/example/*.java src/main/java/org/example/*/*.java

java -cp out org.example.Main
```

## Принятые решения

### 1. Value Objects для полей книги

Вместо примитивных типов (`String`, `int`) использованы классы `Title`, `Author` и `Year`. 

**Почему:**
- Валидация инкапсулирована внутри объектов (название и автор не могут быть пустыми, год - положительным и не будущим)
- Исключена возможность создания некорректной книги
- Код стал типобезопасным - невозможно перепутать параметры при вызове методов

### 2. Кастомные исключения

Созданы отдельные классы исключений: `DuplicateBookException`, `BookNotFoundException`, `InvalidInputException`, `InvalidSortException`.

**Почему:**
- Единая обработка всех ошибок в главном цикле через `catch (RuntimeException e)`
- Понятные и точные сообщения для пользователя
- Код легче читать и поддерживать

### 3. Хранение в оперативной памяти через HashMap

Данные хранятся в `HashMap<Integer, Book>`. Использование H2 (в том числе в in-memory режиме) было осознанно исключено.

**Почему:**
- Прямое указание в ТЗ: *"только в оперативной памяти (не нужно сохранять в файл или БД)"*
- `HashMap` обеспечивает O(1) для добавления, удаления и поиска по ID
- Отсутствие внешних зависимостей упрощает запуск и проверку кода
- Для 5 команд консольного приложения база данных была бы архитектурно избыточной

### 4. Record для модели Book

Использован `record Book` (Java 14+).

**Почему:**
- Меньше шаблонного кода (автоматически генерируются конструктор, геттеры, equals, hashCode)
- Неизменяемость данных по умолчанию
- Код лаконичнее и читаемее

### 5. Разделение на слои

Проект разделён на пакеты: `model`, `service`, `exception`, `valueobjects`.

**Почему:**
- Чёткое разделение ответственности
- Легко тестировать бизнес-логику отдельно от консольного ввода-вывода
- Готовность к расширению (например, замена консольного интерфейса на веб)
